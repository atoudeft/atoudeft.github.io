Agrégation/Composition

Dans la vie réelle, on peut assembler des objets pour construire des objets plus complexes. Ainsi, on peut assembler une carte mère, des barrettes de mémoire, un disque dur, … pour construire un ordinateur. On peut assembler des briques pour construire un mur, puis assembler des murs, des portes et des fenêtres pour construire des chambres, puis assembler des chambres pour construire des appartements, etc…

De manière similaire, dans un programme orienté objet, on peut assembler des objets pour construire des objets plus complexes. Il existe 2 concepts très similaires pour cela : l’agrégation et la composition.

Dans l’agrégation, les objets qu’on assemble pour construire un objet complexe sont les agrégats tandis que l’objet complexe construit est l’agrégé.

Dans la composition, les objets qu’on assemble pour construire un objet complexe sont les composants tandis que l’objet complexe construit est le composé.

La différence entre l’agrégation et la composition est reliée à la durée de vie des objets : dans l’agrégation, les agrégés peuvent exister sans l’agrégat. Donc la destruction de ce dernier n’entraine pas la destruction des agrégats. Mais, dans la composition, les composants ne peuvent pas exister sans le composé. La destruction du composé entraine donc la destruction des composants.

Voici quelques exemples :

Une bibliothèque contient des livres. Un livre peut exister sans la bibliothèque. Il y a donc une relation d’agrégation entre bibliothèque et livre.

Dans une école, il y a des étudiants. Un étudiant peut exister sans être rattaché à une école. C’est donc une relation d’agrégation entre école et étudiant.

Une maison contient des chambres. La destruction de la maison entraine la destruction des chambres (une chambre ne peut pas exister en dehors d’une maison). Il y a donc une relation de composition entre maison et chambre.

Dans une banque, des opérations bancaires sont effectuées sur des comptes. Plusieurs opérations bancaires peuvent être rattachées à un compte. Mais, une opération bancaire ne peut pas exister sans être rattaché à un compte. Il y a donc une relation de composition entre compte bancaire et opération bancaire.

Dans des langages tel que Java, il n’y a pas de différence notable dans l’implémentation de l’agrégation et de la composition. Nous ne ferons donc pas de distinction entre ces concepts dans la suite.

L’agrégation/composition implémente la relation « possède un » (has a ou hasA). Elle se traduit par la présence dans l’agrégé/composé d’attributs de type de l’agrégat/composant.

La plupart du temps, l’agrégé/composé est appelé à collaborer avec ses agrégats/composants.

Voyant un exemple :

Nous avons une classe qui représente des compteurs :

public class Compteur {

private int valeur;

//...

public int getValeur() {

return valeur;

}

public void incrementer() {

valeur++;

}

}

Une voiture possède un compteur. Et lorsque la voiture avance, son compteur augmente :

public class Voiture {

private Compteur compteur;

//...

public void avancer(int distance) {

for (int i=0;i<distance;i++)

compteur.incrementer();

}

public int getValeurCompteur() {

return compteur.getValeur();

}

}

Voici un autre exemple :

La classe Etudiant représente des étudiants :

public class Etudiant {

private double note;

//...

public double getNote() {

return note;

}

}

Un cours-groupe contient des étudiants inscrits à un même cours. On trouve donc dans un objet de type cours-groupe un attribut de type « liste d’étudiants » qui peut être implémenté sous-forme d’un tableau (en attendant de voir d’autres structures de données, comme les vecteurs) :

public class CoursGroupe {

private Etudiant[] listeEtudiants;

//...

public double getMoyenne() {

double somme = 0, moyenne;

for (int i=0;i<listeEtudiants.length;i++)

somme = somme + listeEtudiants[i].getNote();

moyenne = somme/listeEtudiants.length;

return moyenne;

}

}

Voici un dernier exemple :

Dans un cours donné, un étudiant possède plusieurs évaluations (intra 1, intra 2, TP1, …). On trouve donc dans un objet de type étudiant un attribut de type « liste d’évaluations » qui peut être implémenté sous-forme d’un tableau.
